# Customization Guide

Use this repo as a static client workspace starter. Keep the public repo fictional, then customize private forks for real client delivery.

## 1. Replace Fixture Data

Edit `src/data/clientWorkspace.ts`:

- `clientLabel` and `workspaceLabel` for the workspace name.
- `milestones` for the client journey or service package.
- `approvalState` and `progress` for current milestone status.
- `files` and `requests` for placeholders, access needs, copy needs, and review assets.
- `questions` for client-facing clarifications.
- `decisions` for scope calls, approvals, and next steps.
- `closeoutChecklist` for final handoff requirements.

## 2. Adjust The Theme

The starter uses a small theme object in `sampleWorkspace.theme`. Update `accent`, `accent2`, `ink`, `soft`, and `warm` for a private fork while preserving contrast.

## 3. Keep Exports Useful

The JSON and Markdown exports are generated by `src/exporters/statusReport.ts`. Add fields only when they are useful in both:

- client update summaries,
- internal handoff notes,
- closeout reports,
- reusable templates.

Run `npm run test:smoke` after changing exporter logic.

## 4. Update Client Briefs

Copy `docs/client-brief-template.md` into a private project folder and fill it in before replacing fixture data. Keep this public template generic.

## 5. Refresh Screenshots

If the UI changes, refresh `docs/demo-screenshot.png` with the local app running so the README reflects the current starter.

## 6. Stay Static By Default

This repo intentionally has no backend, auth, external service calls, credentials, or real data. If a private fork needs integrations, add them there with a separate data-handling review.
